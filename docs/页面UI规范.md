## 1. 注意事项

- **不要过度蔓延需求**：严格按照描述要求实现，不要添加额外的功能或优化
- 保持代码风格与现有项目一致
- 保存UI风格与现有项目一致
- 要适配浅色和深色主题风格
- 如果有公共组件，则放到src\components里：如有需要直接引用这里面已经有的封装组件（包含列表，设备树，等等）
- 如果功能存在配置参数，参数要清晰，易于后续调整

## 2. 新页面生成布局要求

使用GenericPage.vue生成页面

- **页面结构**：
  - 页头 (Header)
  - 页体 (Content)
- **页头功能**：
  - 右侧显示"页面说明icon“
  - 点击按钮弹出对话框显示当前功能的说明文档（为项目全局统一的，交互完全一样，参考已有页面实现方式）

### 全局文档体系（三类，分工明确）


| 类型           | 给谁看                    | 存放路径                                                           | 入口位置                                 |
| ------------ | ---------------------- | -------------------------------------------------------------- | ------------------------------------ |
| **页面说明**     | 产品/研发/测试/运维，理解**整页**   | `public/docs/pages/`（路由路径命名，含子目录如 `policy-change/nat-open.md`） | 页头 `PageHeader` 右侧「页面说明」             |
| **弹窗「原型说明」** | 同上，理解**单个业务弹窗/抽屉**     | `public/docs/pages/dialogs/{稳定英文短名}.md`                        | 弹窗标题栏右侧图标（公共组件 `DialogPrototypeDoc`） |
| **功能变更记录**   | 责任人做版本追溯，**不面向最终用户弹窗** | `docs/feature-changelog/{与页面或路由对应的文件名}.md`                     | 无 UI；随发版/需求变更人工或脚本追加                 |


说明：

- 「**自动梳理**」指文档**应覆盖**的维度（功能、业务逻辑、UI 等），由人工撰写或 AI/脚本**生成后写入对应 `.md` 并审核入库**；**不是**指前端在运行时根据代码自动生成说明。
- 此前若仅有规范、业务弹窗未接 `DialogPrototypeDoc`，属于**落地未完成**，需按上表补齐。

### 页面说明文档

- **文档格式**：Markdown
- **存放位置**：`public/docs/pages/` 目录
- **命名规范**：使用路由路径作为文件名（去掉前导斜杠），如：
  - `docs/pages/policy-analysis.md`
  - `docs/pages/business-performance.md`
  - `docs/pages/alert-center.md`
  - 等等
- **加载方式**：实时读取对应的 Markdown 文件内容并在弹窗中渲染显示
- **文档内容**：须达到「**前后端可直接对照开发**」的深度，而非仅有口号式概述。最低要求见下节 **§2.1**；页面级文档建议包含的子章节见 **§2.1.1**，弹窗级见 **§2.1.2**。
  - **版本与维护追溯**：发版/需求变更后的时间线、责任人备注，以 `**docs/feature-changelog/`** 为准；页面/弹窗说明正文**可保留**文末极简「文档修订记录」（可选），与功能变更长文分离。

### §2.1 说明文档深度标准（开发可用）

**原则**

- **单一事实来源**：该路由或该弹窗的交互、字段、接口约定，以对应 `.md` 为准；与 PRD 冲突时以**已评审定稿**为准并同步改文档。
- **先边界后细节**：先写范围/非目标，再写数据模型与接口草稿，再写 UI 与异常。
- **可测试**：关键规则写成**可写成用例**的陈述（Given/When/Then 或表格）。

**页面级文档（`public/docs/pages/...`）建议结构（§2.1.1）**


| 章节       | 内容要点                                             |
| -------- | ------------------------------------------------ |
| 路由与依赖    | 路径、`name`、页头/页体组件、依赖的公共组件                        |
| 范围与非目标   | 本页负责什么、**明确不做**什么（避免前后端各自发挥）                     |
| 信息架构     | 区域划分、用户主路径（从进入到离开）                               |
| 筛选与列表    | 筛选维度、默认值、组合逻辑、分页、排序（若有）                          |
| 主数据模型    | 列表行字段表：字段含义、类型、枚举、展示规则、是否可排序                     |
| 状态与聚合    | 业务状态定义、多设备/多行如何聚合成「未完成」等                         |
| 接口契约（草稿） | `GET/POST` 路径、Query/Body 字段、响应结构；**未定时**标注「联调待定」 |
| UI/交互细则  | 空态、加载、按钮禁用条件、表格行高/固定列、主题                         |
| 原型与实现    | 与当前代码/设计稿差异、Mock 行为说明                            |


**弹窗级文档（`public/docs/pages/dialogs/...`）建议结构（§2.1.2）**


| 章节         | 内容要点                                               |
| ---------- | -------------------------------------------------- |
| 入口与前提      | 从哪打开、是否需要选中行、权限（若有）                                |
| 布局分区       | 左/右、上/下、各区块职责                                      |
| 表单字段表      | 每行：**字段标签、对应前端 state/接口字段、类型、必填、默认值、校验、联动（显示/禁用）** |
| 操作与状态机     | 主按钮顺序、点击前置条件、提交后界面如何变化                             |
| 接口契约（草稿）   | 提交 Body、关键响应字段、错误码或业务错误文案                          |
| 列表/子表格（若有） | 列定义、行操作、与主表关系                                      |
| 异常与边界      | 网络失败、并发、只读模式、超长文本                                  |
| 关联文档       | 父页面说明、子弹窗 `docKey`                                 |


> **提示**：生成文档时可对照 `src/types/`、页面 `ref` 命名与现有 `mock`；接口未定时用「草稿」二字，避免与真实 Swagger 混淆。

**页面级正文章节顺序**：建议按 **§2.1.1** 表格组织；**弹窗级**按 **§2.1.2** 组织。文末可附极简修订记录，详细版本历史见 `docs/feature-changelog/`。

### 弹窗「原型说明」（全局）

与 **§2「页面说明文档」** 并列：**页面级说明**对应**整页路由**；本节对应**业务弹窗 / 抽屉等浮层**，存储格式与 `public/docs/pages/` 约定一致，仅扩展命名规则。

#### 目标

- 在每个**业务弹窗**（含 `ElDialog`、全屏弹层、抽屉等）右上角提供统一的**「原型说明」**入口（图标按钮）。
- 点击后打开**仅针对当前弹窗**的说明浮层，展示该弹窗的**功能、业务逻辑、UI 与交互**等，便于产品、研发、测试对齐。
- 说明内容以 **Markdown 文件**维护，随功能迭代**人工修订**（发版或需求变更后由责任人更新），并在文档内保留**维护记录**，形成可追溯的「**单弹窗一页说明**」（即：一个弹窗对应一个 `.md` 文件，变更通过 Git 与文内维护记录可追溯）。

#### 交互与 UI（全局一致）


| 项           | 约定                                                                                                                             |
| ----------- | ------------------------------------------------------------------------------------------------------------------------------ |
| **位置**      | 弹窗 **header 标题栏右侧**（与标题同一行），位于关闭按钮左侧；若弹窗无标准 header，则在**内容区右上角**固定一块与标题对齐的区域。                                                   |
| **形态**      | **仅图标**（建议 `InfoFilled` / `QuestionFilled` / `Document` 等与页头「页面说明」可区分或统一，全项目一致）。                                               |
| **Tooltip** | 悬停文案统一为：**「原型说明」**（全项目统一）。                                                                                                     |
| **点击**      | 打开**次级对话框**（或 `ElDrawer`），标题建议：**「说明 — {弹窗业务简称}」**。                                                                            |
| **内容区**     | 与「页面说明」一致：**实时拉取 Markdown 文本**展示（实现可参考 `PolicyAnalysis` 页：先 `fetch` 再展示；是否用 `<pre>` 展示纯文本或后续改为 Markdown 渲染，**全项目与页面级说明保持一致**）。 |
| **加载与空态**   | 与页面级一致：加载中、404/失败时显示占位文案（如「暂无说明」）。                                                                                             |
| **主题**      | 适配浅色 / 深色主题，与现有 `ElDialog` 一致。                                                                                                 |


#### 存储约定（在 §2「页面说明文档」基础上扩展）

- **文档格式**：Markdown（`.md`）
- **存放位置**：`public/docs/pages/` 目录下（**不另起根目录**，便于统一静态资源与部署规则）
- **命名规范**：
  - **路由页面**（同 §2）：`{路由路径去前导斜杠}.md`，例如 `policy-analysis.md`。
  - **弹窗说明**（扩展）：使用子目录区分，避免与路由文件名冲突：  
  `**public/docs/pages/dialogs/{稳定英文短名}.md`**  
    - `{稳定英文短名}`：全站唯一，建议 `**业务域-功能-弹窗`** 式 kebab-case，例如：`policy-open-adjust.md`、`security-policy-batch-import.md`。  
    - 同一弹窗迭代时**不随意改名**；产品重命名功能时同步重命名文件并全局替换引用。
- **加载方式**：运行时 `fetch` 对应路径（如 `/docs/pages/dialogs/policy-open-adjust.md`），在说明浮层中展示。
- **版本管理**：说明文档纳入 Git，与业务代码**同库维护**。

弹窗说明文档须满足 **§2.1**、**§2.1.2** 的深度要求；修订方式：需求或实现变更后，责任人据代码与产品定稿更新；AI 辅助生成须**人工审核**后入库。

#### 工程实现要点

- **公共组件**：`src/components/DialogPrototypeDoc.vue`  
  - **标准**：传 `docKey`（对应 `public/docs/pages/dialogs/{docKey}.md`）+ `business-short-name`（次级弹窗标题为「说明 — {简称}」）。  
  - **例外**：页头若需读取**非** `dialogs/` 下文件（历史路径），可传 `doc-url`（如 `/docs/pages/policy-change-open.md`），仍复用同一套加载与展示逻辑。
- **接入方式**：在 `ElDialog` 上使用 `**#header` 插槽**（与 Element Plus 约定一致），在标题与关闭按钮之间放置 `DialogPrototypeDoc`；无标准 header 时在内容区右上角固定等价入口（见上表「位置」）。
- **路径**：`docKey` 或模块内常量集中定义，**禁止**多处硬编码同一文档完整 URL。

#### 与「页面说明」的边界


| 维度     | 页面说明（§2）                         | 弹窗原型说明（本节）                                  |
| ------ | -------------------------------- | ------------------------------------------- |
| 入口位置   | 页面 `PageHeader` 右侧按钮             | 弹窗标题栏右侧图标                                   |
| 文档对应   | 当前**路由**                         | 当前**弹窗**（稳定 `docKey`）                       |
| 典型路径示例 | `/docs/pages/policy-analysis.md` | `/docs/pages/dialogs/policy-open-adjust.md` |


二者可同时存在：用户可先读页面级说明，再在具体弹窗内读弹窗级说明。

## 3. 页头右上角功能入口交互规范（全局）

### 3.1 适用范围

- **适用页面**：所有存在业务功能入口的页面（例如某一级菜单下包含多个常用子功能入口的场景）。
- **适用区域**：统一使用 `PageHeader` 的右上角区域（`actions` 插槽）展示功能入口。

### 3.2 显示原则（固定显示，不因点击隐藏）

- **固定显示**：同一业务域的一组功能入口在页头右上角**始终固定显示**。
- **不隐藏**：点击任意入口并跳转后，其余入口**仍保持可见与可点击**，不做隐藏/移除。
- **一致性**：同一业务域下所有子页面必须保持**同一组入口及顺序**，避免页面间入口缺失导致用户迷失。

### 3.3 状态反馈（选中态）

- **当前入口高亮**：当路由处于某个子功能页时，对应入口展示高亮样式（例如 `primary`），其它入口保持常态（例如 `plain`），但不隐藏。
- **总览页规则**：当处于业务域总览页（例如 `/policy-change`）时，默认不强制高亮某个入口（如需高亮规则可在具体业务中定义）。

### 3.4 路由与信息架构建议

- **业务域与入口组对应**：每个一级菜单对应一个业务域；常用子功能以“入口组”形式沉淀在页头右上角。
- **推荐路由结构**：
  - 业务域总览：`/xxx`
  - 子功能页面：`/xxx/a`、`/xxx/b`、`/xxx/c` ...
- **复用要求**：业务域总览页与所有子功能页应复用同一份入口配置，确保入口组统一与可维护。

### 3.5 视觉与布局要求

- **布局**：入口右对齐，水平排列，按钮间距一致；小屏下优先保持顺序，必要时缩小按钮尺寸/文字而非改变入口集合。
- **文案**：使用简洁动作型短语（例如“策略开通”“策略修改”）。
- **样式**：入口按钮不抢占主标题视觉焦点，建议使用轻量按钮风格（例如小尺寸按钮），并通过选中态区分当前入口。

### 3.6 开发实现要求（约束）

- **统一封装**：为每个业务域封装入口渲染（组件或配置驱动），供该业务域所有页面复用。
- **禁止做法**：
  - 禁止在不同子页面中维护不同的入口集合（导致入口不一致）。
  - 禁止点击入口后通过条件渲染将其它入口从 DOM 中移除或隐藏。

## 4. 列表通用 UI 规范（分页 / 高度 / 空行显示）

### 4.1 分页与行数展示（全局通用）

- **固定每页行数**：列表按分页设置显示固定行数（例如 10 条/页）。即便当前页真实数据不足该行数，也需补齐空白行，使表格在视觉上始终保持整页高度一致（例如每页固定 10 行）。
- **空行的渲染规则**：补齐的空行与真实数据行在表格结构上一致，但所有字段为空。空行不显示占位符或操作项，见 4.3。

### 4.2 自适应屏幕高度与滚动（全局通用）

- **自适应可视区域高度**：表格容器高度由布局自适应（如 flex:1），每页固定行数，保证整页视觉高度统一。大屏下当前页行数全部可见；小屏或窗口高度不足时，表格区域无法容纳当前页全部行数时，仅在表格内部出现纵向滚动条，不改变整页布局高度。
- **纵向滚动行为**：仅列表主体区域出现纵向滚动条（行内容上下滚动），分页栏、表头等保持在列表区域外，不随内容滚动。
- **横向滚动行为**：在有横向溢出时固定显示横向滚动条，高度固定为可视区域，横向滚动条始终出现在视口底部，不再依赖当前行数。

### 4.3 空行显示规范（全局通用）

- **字段列**：仅在有真实数据时渲染单元格内容（文本、标签、图标等）。对于补齐的空行，字段列保持空白，不显示「-」、0、「无数据」等占位文案。
- **操作列**：仅对真实数据行渲染操作按钮（详情、编辑、更多等）。空行的操作列不渲染任何内容，完全空白。

### 4.4 表格列规范（全局通用）

- **内容列字号统一**：除表头外，表格所有内容列的字号应统一（建议 12px），避免某列内容字号明显大于其他列。
- **操作列宽度自适应**：操作列使用 `min-width` 而非固定 `width`，使列宽根据显示的按钮数量与文案自适应；建议最小宽度约 100px（可按实际按钮数量调整）。
- **操作列左右边距一致**：操作列单元格的左右内边距应一致（如均为 12px），保证首字到左边距与末字到右边距相等；若为固定右侧列，需覆盖 Element Plus 默认样式以保持左右边距一致。

### 4.5 列设置与主表联动（全局通用）

适用于主内容区**数据表格**的列配置；**弹窗、抽屉内表格**不参与同一套列设置（除非页面单独约定）。

**配置模型（摘要）**

- 每列至少包含：`key`、`label`、`visible`、`order`、`fixed`（固定仅作用于**整列**）。
- **多字段列**可配置 `fields[]`（子块显隐）；列显示时**至少保留一个子字段为显示**，禁止出现「列显示但子字段全关」的空列。
- 勾选列、序号列等是否参与列设置由产品约定；**操作列**可纳入列设置（如默认右固定）。

**与列表的联动**

- 主表仅渲染「`visible` 为真且按 `order` 排序」的列；单元格内块由子字段显隐控制（无 `fields` 则整列显即内容显）。
- **当前列配置状态为单一数据源**：在面板中修改列显隐、顺序、固定、子字段后，表格应**即时**反映，不依赖点击「保存」才在当次会话生效。

**列设置面板内操作（摘要）**

| 能力 | 说明 |
| --- | --- |
| 列表样式（下拉） | 切换时加载对应模版快照到当前列配置，表格随之更新。 |
| 重置 | 恢复为代码内建的默认列配置；若实现「默认列表」本地持久化，重置时应一并清除该覆盖。 |
| 批量 / 行内 | 对勾选列或单行：隐藏、显示、置顶、置底、上移、下移；行内可含列显隐、子字段显隐、固定三态（左 / 不固定 / 右）。 |
| 另存为 | 输入新名称，**新建**一条列表样式（模版），内容为当前列配置快照；持久化后通常选中该新模版。 |
| 保存（建议置于「另存为」右侧） | 将**当前列配置快照**写入**当前下拉选中的**列表样式：更新该模版内容并持久化。**不新建**模版。 |

**默认列表持久化（推荐）**

- 用户选中「默认列表」并调整列后，点击「保存」应将当前快照写入**默认列表专用存储**（如独立 `localStorage` 键），使**刷新或下次进入页面**仍能恢复其对默认列表的修改。
- 另存为产生的是**具名新模版**，写入自定义模版存储；与默认列表覆盖区分存放。
- 加载时建议先用代码内建默认列定义初始化，再与默认覆盖数据做**合并**（便于新版本增加列、子字段时兼容旧数据）。

## 5. 表单：只读输入 + 点击唤起选择（全局默认）

当表单项需要从**列表/弹窗**中选择结果，且不希望单独放置「选择」按钮时，**默认采用本格式**，与业务弹窗（单选列表、确定/取消）组合使用。

### 5.1 模式标识与适用范围


| 项        | 说明                                           |
| -------- | -------------------------------------------- |
| **模式名称** | 只读选择器（Readonly Picker Input）                 |
| **适用**   | 选项需独立弹窗/抽屉承载（说明、搜索、单选等），表单项旁**不**再放独立「选择」主按钮 |
| **不适用**  | 简单枚举、少量固定项 → 优先用 `Select` 或内联下拉，不必强行改为本模式    |


### 5.2 交互（必须）

- **展示**：单行 `Input`，`readonly`，展示当前已选文案；未选时为空。
- **打开**：用户**点击整个输入区域**即打开**既有**选择弹窗（与原先点「选择」为**同一套**逻辑与数据源，仅更换触发入口）。
- **禁用**：与表单项或业务禁用一致：不可点击、不打开弹窗。
- **占位符**：统一表达「点击选择」，建议使用模板 `**点击选择{资源名称}`**（示例：`点击选择可合并策略`、`点击选择置顶策略`）。

### 5.3 视觉（必须）

- **指针**：可交互时输入区域为 `**cursor: pointer`**（外层容器与内部 `input` 建议一致，避免仅外框可点）。
- **只读语义**：使用组件 `readonly`，避免用户误以为可键盘输入。

### 5.4 与弹窗的契约

- 输入框只负责**展示 + 触发**；弹窗负责列表与确认；确认后回写绑定值，输入框自动更新展示。
- **禁止**为同一选择再增加一个独立「选择」按钮作为唯一入口（若需「清空已选」，可用次要图标或表单项级操作，不作为主流程按钮）。

### 5.5 全局默认配置（可复制到项目配置 / 设计令牌）

以下为**推荐默认**，新需求有类似「从弹窗选一项填到输入框」时，按此调用；实现层可用 Element Plus `ElInput` 等，字段语义保持一致即可。

```yaml
# 表单 - 只读选择器（点击打开弹窗）
form.readonlyPickerInput:
  id: readonly-picker-input
  description: 只读输入框 + 点击打开既有选择弹窗，无独立「选择」按钮
  input:
    readonly: true
    cursor: pointer
    placeholderPattern: "点击选择{label}"   # label 由业务传入，如：可合并策略、置顶策略
  trigger:
    event: click
    opens: existingDialog                   # 复用已有选择弹窗，不替代弹窗业务逻辑
  state:
    disabledFollowsForm: true
```
